## Discovery / process overhead (CTest wall vs GoogleTest body time)

The unit binary `lci_tests` registers **1851 separate ctest entries**, each a fresh process. Summed CTest per-entry wall is **28.0s**, but the same tests run in ONE process (`--gtest_output=json`) report only **5.6s** of test-body time.

=> **22.4s (79.9% of unit entry wall, ~12.1 ms/entry)** is process spawn + per-process global init/discovery overhead, NOT test-body work. Wall time must not be attributed entirely to test bodies.

## Candidates for follow-up slices (measured only)

### S2 — CTest discovery granularity / parallel races

- gtest_discover_tests per-entry spawn overhead = 22.4s (79.9% of unit entry wall) across 1851 single-test processes (~12.1 ms/process). Coarser discovery granularity (fewer ctest entries per binary) directly reclaims this.
- RUN_SERIAL entries force sequential execution and cannot overlap the -j pool: `lci_integration_suite` (272.9s), `lci_real_project_suite` (26.9s). On a 12-way box these gate the tail of the wall clock.
